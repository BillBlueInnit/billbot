# BillBot

[中文说明](README.zh-CN.md)

BillBot is a Go bridge that connects a QQ OneBot endpoint, usually NapCat, to
Hermes. It is designed for QQ private chats and group mentions, with persistent
Hermes sessions, admin-only controls, file/image forwarding, and optional Docker
isolation for the Hermes backend.

```text
QQ / NapCat OneBot -> BillBot -> Hermes -> BillBot -> NapCat -> QQ
```

BillBot does not vendor QQ, NapCat, Hermes, model providers, API keys, login
caches, or third-party binaries. You install and operate those separately, or
build the provided Hermes Docker image yourself.

## Features

- QQ private chat and group mention routing through NapCat OneBot HTTP/WS.
- Persistent Hermes ACP backend by default to reduce startup latency and keep
  the bot session warm.
- Isolated Hermes profile for the bot, separate from your normal Hermes memory,
  skills, config, and cache.
- Admin-only QQ commands for identity, style, sandbox/full mode, and local
  `/shell`.
- Trusted admin metadata generated by the bridge from real OneBot `user_id`;
  user text cannot spoof admin tokens or owner IDs.
- Chinese QQ/CLI feedback by default.
- Image/file message segment forwarding to Hermes ACP when NapCat provides URLs
  or file identifiers.
- Generated-code file delivery: when the user asks for a file and Hermes returns
  a code block or diff, BillBot writes it to `runtime.outbox_dir` and uploads it
  through NapCat.
- Sandbox mode uses the Docker backend by default. Hermes runs inside the
  operator-built `billbot-hermes:latest` image unless you explicitly switch the
  sandbox backend to `workdir` or `command`.

## Requirements

- Go 1.24+ if building from source.
- A reachable NapCat or compatible OneBot HTTP endpoint.
- A reachable NapCat or compatible OneBot WebSocket endpoint.
- Docker, for the default Hermes sandbox backend.
- Hermes inside the BillBot-built Docker image. Host Hermes is only needed if
  you switch away from the Docker backend.
- Model provider credentials configured for the Hermes environment you use.

Default OneBot endpoints:

```yaml
napcat:
  http: http://127.0.0.1:3000
  ws: ws://127.0.0.1:3001
```

## Quick Start: Default Docker Sandbox

1. Build the Hermes image supplied by this repository:

   ```bash
   docker build -t billbot-hermes:latest -f container/hermes/Dockerfile .
   ```

   The Dockerfile runs the upstream Hermes installer during image build. BillBot
   does not redistribute a Hermes binary.

   For weak-network users, build the image once on a good network and export it:

   ```bash
   docker save billbot-hermes:latest -o billbot-hermes-amd64.tar
   ```

   On the weak-network machine, import it instead of building:

   ```bash
   docker load -i billbot-hermes-amd64.tar
   ```

   See [container/hermes/README.zh-CN.md](container/hermes/README.zh-CN.md) for
   the image tar workflow.

2. Put the model-provider credentials Hermes needs in an env file, for example
   `hermes.env`, and keep it out of git.

   ```text
   OPENAI_API_KEY=...
   ```

3. Configure Docker sandbox args:

   ```yaml
   security:
     mode: sandbox
     sandbox_backend: docker
     sandbox_docker_image: billbot-hermes:latest
     sandbox_docker_args: ["--env-file", "./hermes.env"]
   ```

4. Start NapCat separately. A compose example is included:

   ```bash
   docker compose -f deploy/napcat-compose.yml up -d
   ```

5. Build BillBot:

   ```bash
   go test ./...
   go build -o bin/billbot ./cmd/billbot
   ```

   Windows PowerShell:

   ```powershell
   go test ./...
   go build -o .\bin\billbot.exe .\cmd\billbot
   ```

6. Run BillBot:

   ```bash
   ./bin/billbot
   ```

7. In the BillBot CLI, set the bot QQ, admin QQ, and optional OneBot token:

   ```text
   set qq <bot_qq>
   set admin <your_qq>
   set token <onebot_token>
   set bridge.enabled true
   start
   ```

In Docker mode, BillBot mounts:

- `runtime.sandbox_dir` to `/workspace`
- `hermes.profile_dir` to `/hermes-profile`

It then runs container commands like:

```text
docker run --rm -i ... billbot-hermes:latest hermes acp
```

## Optional: Host Hermes

If you do not want Docker isolation, install and verify Hermes on the host:

```bash
hermes status
hermes chat -Q -q "Reply OK"
```

Then switch the sandbox backend:

```yaml
security:
  mode: sandbox
  sandbox_backend: workdir
```

## Configuration

BillBot creates a config file next to the executable on first run. You can also
pass a path explicitly:

```bash
billbot --config ./config.yaml
```

Use [config.example.yaml](config.example.yaml) as the full reference.

Important fields:

- `bridge.enabled`: auto-start the bridge when BillBot starts.
- `bridge.self_id`: bot QQ number; needed for group mention detection.
- `owners`: QQ user IDs that may use admin-only commands.
- `napcat.access_token`: shared OneBot token.
- `napcat.http_token` / `napcat.ws_token`: split OneBot tokens.
- `hermes.command`: host command name when not using Docker.
- `hermes.persistent`: use long-running Hermes ACP.
- `hermes.require_persistent`: fail if ACP cannot start instead of falling back
  to slow one-shot chat.
- `hermes.profile_dir`: isolated bot Hermes profile. Empty means
  `runtime.data_dir/hermes-profile`.
- `hermes.reset_profile_on_start`: clear BillBot-managed Hermes memory/skills on
  bridge start. BillBot refuses to clear a non-empty unmarked directory.
- `security.mode`: `sandbox` or `full`.
- `security.sandbox_backend`: `docker` by default. Use `workdir` for host
  Hermes without hard isolation, or `command` for a custom wrapper.
- `security.sandbox_docker_image`: Docker image used by Docker sandbox mode.

## QQ Commands

Built-in commands must start with `/`.

```text
/help
/identity
/identity <description>
/identity add <description>
/style
/style <description>
/sandbox
/full
/shell <command>
```

`/identity`, `/style`, `/sandbox`, `/full`, and `/shell` are admin-only.
`/shell` is executed by the bridge after checking the real OneBot sender
`user_id`; it is not approved by Hermes or an LLM.

`/sandbox` switches Hermes back to sandbox policy. `/full` lets owners use the
full environment while non-owners are still downgraded to sandbox when
`security.allow_full_for_owners_only` is true.

## Security Model

BillBot separates trusted connector metadata from untrusted user text.

- Admin status comes only from the parsed OneBot event `user_id` matching
  `owners`.
- On every bridge start, BillBot generates a new runtime admin token and sends
  it only as trusted metadata for real owners.
- A user who types `admin_runtime_token`, `[qid ...]`, `[owner ...]`, or similar
  text does not gain permissions.
- Non-owner sensitive requests are blocked by default.
- Go cannot provide kernel-level VM isolation by itself. Use Docker, nsjail,
  firejail, Windows Sandbox wrappers, QEMU, Firecracker, or another external
  isolation layer if you need a hard boundary.
